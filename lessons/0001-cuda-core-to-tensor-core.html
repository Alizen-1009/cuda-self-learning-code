从 CUDA Core 到 Tensor Core

一条 mma.sync 指令到底做了什么——以及为什么它感觉像黑盒。

这节课的唯一目标 (tangible win) 读完后，你能拿起任何一条 mma.sync 的 PTX，逐字段说出它的含义，并解释为什么"没有任何一个线程持有一整行数据"。这是 wgmma(Hopper)、tcgen05/TMMA(Blackwell) 和整个 CUTLASS 的共同地基——搞懂这一条，后面所有 tensor core 的黑盒都会开始透明。

1. 心智跃迁：从"标量 FMA"到"一条指令一个矩阵"

你写 CUDA core kernel 时的世界观是：一个线程执行一次 fma（c += a*b），32 个线程凑成一个 warp 并行地各算各的。矩阵乘就是每个线程负责输出矩阵的若干个元素，自己拉一条 K 维内积。

Tensor Core 把这个世界观整个换掉。它不是"更快的标量单元"，而是一个矩阵单元：一条指令直接算完一整块小矩阵的乘累加 

D = A · B + C

其中 A、B、C、D 是小矩阵而不是标量。代价是：这条指令不属于单个线程，而是整个 warp（32 线程）协作发起的一条指令。这正是它让人困惑的根源——它长得像一条普通 PTX，语义却是集体行为。[1]

2. 解剖一条真实的 mma.sync

这是 Ada/Ampere 上最常见的一条 FP16 tensor core 指令（你 4060ti 上就能跑）：

mma.sync.aligned.m16n8k16.row.col.f32.f16.f16.f32
    {d0,d1,d2,d3}, {a0,a1,a2,a3}, {b0,b1}, {c0,c1,c2,c3};
字段含义
mma.syncwarp 级同步矩阵乘累加。sync = 整个 warp 必须一起执行、执行完隐式同步。
.aligned保证 warp 内 32 线程全部活跃且对齐参与（否则行为未定义）。
.m16n8k16本条指令算的 tile 形状：M=16, N=8, K=16。即 A 是 16×16、B 是 16×8、C/D 是 16×8。
.row.colA 按 row-major、B 按 col-major 解读（决定 fragment 摆放）。
.f32.f16.f16.f32依次是 D、A、B、C 的数据类型：输入 A/B 是 f16，累加 C/D 是 f32（混合精度）。
{d0..d3} 等操作数是寄存器列表——每个线程贡献自己那几份 fragment，不是内存地址。
读法口诀 mma.sync 后面永远是三段：形状(mXnYkZ) → 布局(row/col) → 类型(D/A/B/C 四个)。看到任何 mma 指令，按这三段拆，一定拆得开。

3. 关键洞察：fragment 分散在 32 个线程的寄存器里

这就是"黑盒感"的核心。这条 m16n8k16 指令要处理的数据量是：

矩阵总元素÷ 32 线程每线程寄存器
A16×16 = 256 个 f168 个 f16{a0,a1,a2,a3} = 4×b32
B16×8 = 128 个 f164 个 f16{b0,b1} = 2×b32
C/D16×8 = 128 个 f324 个 f32{c0..c3} = 4×b32

所以：没有任何一个线程持有一整行或一整列。矩阵被硬件规定的、非直觉的方式"打碎"分布到 32 个线程的寄存器中。你不能自己随便摆——硬件只认这一种 fragment 布局。[2]

4. 于是 ldmatrix 出现了

既然 fragment 布局是硬件钦定且反直觉的，那么"把数据从 shared memory 搬进寄存器并恰好摆成这个布局"就是个麻烦活。ldmatrix 这条 PTX 指令专门干这件事：一个 warp 协作地从 SMEM 装载一块矩阵，直接落成 mma 要的 fragment 排列。

记住这条数据流水线主干，后面每一代架构都是它的升级版： 

Global ──cp.async──▶ Shared ──ldmatrix──▶ Register ──mma.sync──▶ 累加
                                          (fragment 布局)

Hopper 把 cp.async 升级成 TMA、mma.sync 升级成 wgmma（操作数可直接来自 SMEM，省掉 ldmatrix 一步）；Blackwell 再升级成 tcgen05 + TMEM。骨架不变——所以这一课的心智模型能一路复用。
